# Command Injection

## 개요

- Command Injection은 웹 애플리케이션이 사용자 입력값을 검증 없이 시스템 명령어에 포함시킬 때 발생하는 취약점으로,  
공격자가 서버에서 임의의 명령어를 실행할 수 있는 공격 방식입니다.

---

## Command Injection 연산자

| 연산자 | 설명 | 예시 | 동작 방식 |
|--------|------|------|-----------|
| `;` | 명령어 순차 실행 | 127.0.0.1; ls | 앞 명령 실행 후 뒤 명령 실행 |
| `&&` | 앞 명령 성공 시 실행 | 127.0.0.1 && whoami | ping 성공하면 whoami 실행 |
| `\|\|` | 앞 명령 실패 시 실행 | 127.0.0.1 \|\| whoami | ping 실패하면 whoami 실행 |
| `\|` | 출력 전달 (파이프) | ls \| grep txt | 앞 결과를 뒤 명령 입력으로 전달 |
| `` ` ` `` | 명령어 실행 결과 치환 | `whoami` | 실행 결과를 문자열로 반환 |
| `$()` | 명령어 실행 결과 치환 | $(whoami) | 실행 결과를 변수처럼 사용 |

---

## 핵심 개념
- 사용자 입력값이 OS 명령어에 직접 전달될 때 발생합니다.
- 입력값 검증이 없거나 불충분할 경우 공격이 가능합니다.
- 서버 권한으로 시스템 명령어가 실행됩니다.

---

## 공격 방식 설명
공격자는 입력값에 명령어를 추가로 삽입하여 기존 명령과 함께 실행되도록 합니다.  
주로 명령어 구분자(`;`, `&&`, `|` 등)를 이용하여 추가 명령을 실행합니다.

---

## 예시
- 정상 입력  
  → ping 127.0.0.1

- 공격 입력  
  → 127.0.0.1; ls  
  → 127.0.0.1 && cat /etc/passwd  
  → 127.0.0.1 | whoami  

- 웹 입력 예시  
  → IP 입력창에 `127.0.0.1; ls` 입력 시 서버 파일 목록 출력

---

## 특징
- 서버에서 직접 명령어가 실행되므로 영향도가 매우 큽니다.
- 시스템 정보 유출, 파일 조회/삭제, 권한 상승 등으로 이어질 수 있습니다.
- 입력값 검증이 없으면 비교적 쉽게 공격이 가능합니다.

---

## 대응 방안 (간단)
- 사용자 입력값 검증 및 필터링 수행
- 특수문자(`;`, `&`, `|` 등) 차단
- 시스템 명령어 실행 함수 사용 최소화
- 명령어 실행 시 안전한 API 사용 (예: parameterized 방식)

---

## 정리
- Command Injection은 사용자 입력을 통해 서버에서 임의의 시스템 명령어를 실행하는 공격 방식입니다.
